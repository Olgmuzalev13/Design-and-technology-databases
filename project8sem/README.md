# Django-проект `project8sem`

Данный каталог содержит **корневой Django-проект**, который определяет общую конфигурацию системы автоматизированного управления технологическими процессами (АСУ ТП). В отличие от приложений, проект отвечает не за бизнес-сущности, а за глобальные настройки, маршрутизацию и интеграцию компонентов.

## Различие понятий: project и app

В Django принципиально разделяются два уровня:

* **Проект (project)**
  Отвечает за конфигурацию всей системы в целом.

* **Приложение (app)**
  Реализует отдельный логический модуль или подсистему.

Каталог `project8sem` является именно проектом, а `core` - приложением внутри него.

## manage.py

Файл `manage.py` является **точкой входа** для управления проектом.

Он используется для:

* запуска сервера разработки,
* выполнения миграций,
* создания приложений,
* запуска тестов,
* работы с Django Admin.

Примеры задач, которые выполняются через `manage.py`:

* `runserver`
* `makemigrations`
* `migrate`
* `test`
* `createsuperuser`

Важно:

* файл не редактируется в процессе разработки,
* используется как интерфейс управления проектом.

---

## settings.py

Файл `settings.py` содержит **глобальную конфигурацию проекта**.

Здесь настраивается:

* список подключенных приложений (`INSTALLED_APPS`),
* параметры базы данных,
* middleware,
* шаблоны,
* статические файлы,
* параметры безопасности.

В контексте АСУ ТП:

* здесь задается тип используемой СУБД (PostgreSQL),
* подключаются учебные приложения (например, `core`),
* конфигурируется окружение разработки.

Важно:

* `settings.py` относится ко всему проекту, а не к конкретному приложению,
* логика предметной области сюда не добавляется.

---

## urls.py

Файл `urls.py` отвечает за **маршрутизацию HTTP-запросов**.

Его задача:

* сопоставить URL-адреса с представлениями (views),
* делегировать обработку маршрутов приложениям.

Типовой подход:

* в проекте описываются только верхнеуровневые маршруты,
* маршруты конкретной функциональности выносятся в приложения.

Пример логики:

* проект знает, что `/auth/` или `/employees/` существует,
* приложение `core` знает, что именно происходит по этим адресам.

Важно:

* не размещать бизнес-логику в `urls.py`,
* использовать `include()` для подключения URL приложений.

---

## wsgi.py и asgi.py

Данные файлы обеспечивают **интеграцию Django с сервером приложений**.

Назначение:

* `wsgi.py` используется для классического развертывания,
* `asgi.py` предназначен для асинхронных сценариев.

В рамках учебного проекта:

* эти файлы не требуют изменений,
* используются автоматически при запуске.

## Когда создавать новое приложение

Создание нового Django-приложения оправдано, если выполняется хотя бы одно из условий:

* появляется **новая подсистема** (например, учет оборудования, журнал аварий),
* функциональность логически обособлена,
* предполагается независимое развитие модуля,
* код начинает разрастаться и теряет читаемость.

Примеры отдельных приложений:

* `core` - пользователи и базовая структура,
* `equipment` - технологическое оборудование,
* `monitoring` - сбор и визуализация данных,
* `reports` - отчеты и аналитика.

---

## Когда дописывать функционал в существующее приложение

Дополнение существующего приложения допустимо, если:

* новая логика относится к уже существующей предметной области,
* используется та же модель данных,
* функциональность является расширением, а не отдельной подсистемой.

Примеры:

* добавление полей сотрудника в `core`,
* расширение форм регистрации,
* дополнительные страницы списков и деталей.

Важно:

* избегать превращения одного приложения в «свалку» логики,
* при росте сложности следует выделять функциональность в отдельный app.

## Общие архитектурные рекомендации

* Проект (`project8sem`) должен оставаться минимальным.
* Вся предметная логика выносится в приложения.
* Одно приложение - одна логическая зона ответственности.
* Структура проекта должна позволять масштабирование без переписывания ядра.

Данный подход соответствует промышленной разработке web-систем и может быть использован как основа для курсовых и выпускных квалификационных работ по тематике АСУ ТП.
